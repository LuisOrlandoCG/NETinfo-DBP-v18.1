NETinfo DBP Version 18.1 : Beneficiary Maintenance

Created by Fivi Konstantinou, last modified by 1239-Mythillou Stefani on 19, 11, 2024

End-users can create/view/amend/delete beneficiaries. One beneficiary (individual or company) may have multiple beneficiary accounts connected, each account held by different bank (either domestic or international bank). Beneficiaries are linked to a bank customer thereby allowing multiple end-users (with the associated privileges) to use/view/amend/delete them if/as required (configurable). 

Templates linked to a beneficiary may include information particular to a fund transfer e.g. Transfer Amount, Bank BIC/BIK/Clearing Code, Correspondent Bank details, Ruble related parameters such as KPP/VO/Tax code, Purpose, Charges etc.

Note that the capability is also provided for end-users to view and use (in payments) in mobile native app the beneficiary entries created through the web UI.

Product Main Features / Business Requirements

  1. Beneficiaries are linked to Customer (CIF)

  2. Beneficiaries are linked to templates which enable certain payment related details to be stored for later use

  3. The beneficiary must be connected to at least one template. The template contains information on the beneficiary account and other payment related information.

  4. For the successful creation of a beneficiary the following information must be provided:

    1. Connected customer

    2. Beneficiary name

    3. Beneficiary address

    4. Type of fund transfer

    5. Beneficiary account number

    6. Other compulsory information for the particular type of fund transfer selected e.g. Beneficiary Bank details in case of fund transfer to another bank, either local or international.

  5. The following information may optionally be stored in a Beneficiary:

    1. Beneficiary alias

    2. Beneficiary country

    3. Optional information for the particular type of fund transfer selected e.g. From and To Descriptions, Amount and Currency

    4. ‘Purpose’ and ‘Charges to be paid by’ where it applies

  6. End-users can use and manage (i.e. view / amend / delete) beneficiaries created by them

  7. End-users may restrict other end-users from accessing beneficiaries linked to their primary CIF

  8. In case of multiple end-users being connected to a Customer (CIF), the beneficiaries created by one end-user might (if restricted flag is not selected) be viewed/used/amended/deleted by other end-users connected to the Customer.

  9. Beneficiaries can be exported in PDF.

  10. SCA can be applied when creating / amending a beneficiary

Activity Diagram

The below figure depicts the end-user workflow for creating and managing Beneficiaries. Referring to this, through the ‘Beneficiaries Maintenance’ page, the end-user can:

  • View, delete, amend and export existing beneficiaries (the end-user has created or are not marked as restricted) or

  • Create a new one

  During the process for creating/amending a beneficiary the end-user:

  • Completes any mandatory information (i.e. connected customer, beneficiary name and beneficiary address)

  • Links accounts to the beneficiary by:

    • Linking the beneficiary to a fund transfer type (i.e. Local Bank, SEPA, International, Ruble)

    • Providing mandatory transfer details as required by the selected fund transfer.

Once all compulsory beneficiary information has been entered correctly, then end-user submits and confirms the updated beneficiary. If SCA has been configured, then the associated SCA process is followed.

About the Activity Diagram: The Activity Diagram depicts the main activities in the workflow: boxes indicate end-user activity, decision boxes indicate system decision points and arrow text in brackets indicate major system activity/actions.

Beneficiary Maintenance Page

The figure below (UI/UX - Beneficiary Maintenance), presents the 'Beneficiary Maintenance’ page. Using this page, the end-user can view all beneficiaries to which they have access. The end-user can search for a specific beneficiary using keyword search or filter displayed beneficiaries by Customer.

End-user may select a beneficiary and:

  • View the associated linked accounts using the expand button,

  • Amend the beneficiary details, or

  • Delete the beneficiary

The beneficiary list can be exported in PDF using the associated icon

A disclaimer may appear at the bottom of the page if configured.

Beneficiary Details form

The ‘Beneficiary Details’ form enables the end-user to create and amend beneficiaries. The creation/amendment of a beneficiary follows a three steps process:

  • Input step: End-user inputs information relating to a beneficiary; field validations are performed upon pressing the ‘Submit’ button

  • Confirm step: End-user is displayed with beneficiary information and confirms transaction

  • Complete step: End-user is displayed with the status of the transaction

Beneficiary Details Confirm Page

Referring to the screen below (UI/UX - Beneficiary Details Confirm), end-user is displayed with all information relating to the creation /amendment of the beneficiary. In case where SCA is required, then end-user has to go through the associated SCA process as configured in User Profile.

End-user is prompted to either Confirm or Cancel creation /amendment of the beneficiary.

A disclaimer is displayed if configured

Screen Actions & Event Handling

Object

Event

Actions

Cancel

End-user selects Cancel.

End-user is navigated to Beneficiary Details Input page

Confirm and SCA enabled

End-user selects Confirm and SCA enabled.

A modal appears requesting SCA authentication depending on the configured SCA method. Refer to the SCA document

Confirm

End-user selects Confirm

AND all validations are successful.

End-user is redirected to Beneficiary Details Complete page.

Beneficiary Details Complete

Referring to the screen below (UI/UX - Beneficiary Details Complete), the end-user is displayed with the status of the beneficiary addition and can be redirected to the Beneficiaries Maintenance page by pressing the Done button.

A disclaimer is displayed if available.
