NETinfo DBP Version 18.1 : Ruble Transfers

Created by Fivi Konstantinou, last modified by 1057-Michael Philippos on 06, 11, 2024

End-users can initiate a money transfer in Rubles from one of their own accounts (or from an account they have the relevant privileges) to a beneficiary account held in a Russian Bank. End-users can do this by entering the beneficiary and payment details manually (end-user can select the Bank of interest by searching through the list of banks connected to the network) or retrieving (most of) them from registered beneficiaries or previous fund transfers to the selected beneficiary . The payment instruction can be submitted (to the Core System) for execution in real time or at a future date or as a standing order.

As with any other type of Fund Transfer, Ruble Fund Transfers conforms to the three steps process (i.e. submit, confirm, complete) and certain configuration needs to be done at the Administrator console for its use.

Product Main Features / Business Requirements

  1. Eligible debit accounts are:

    1. Accounts that fall under a Bank Product that has been configured in NETinfo DBP to allow debits (from its accounts) for the particular transaction type

    2. Accounts that end-user has Input privileges (for the particular transaction type) as defined in User Profile.

    3. Accounts under the Customer selected by end-user from the ‘Owner’ field.

  2. Fund transfer details can be filled using any of the following methods:

    1. Manual entry

    2. Automatic retrieval of beneficiary and certain payment details from registered beneficiaries

    3. Automatic retrieval of beneficiary and certain payment details from previous fund transfers to the same beneficiary

  3. The BIK Code is autofilled based on end-user selection from a list of available Banks connected to the associated network

  4. Select the VO Code from a list of available codes

  5. Cross currency transfers are supported with the option to use a negotiated rate. In a cross currency transfer, the currencies of the Debit Account and Transfer Amount differ.

  6. The currency of the transfer amount is always Rubles (defined at installation)

  7. Signatures are supported (configurable).

  8. Calculation of charges is supported (configurable)

  9. SCA functionality is supported (configurable).

  10. Daily limits at customer, account, user and user-account level can be enforced (configurable).

  11. Minimum and Maximum Transaction Limits can be enforced (configurable).

  12. Option to specify different fund transfer descriptions for the sender and beneficiary

  13. Option to add other relevant information: Beneficiary Tax Code and KPP Code

  14. Option to add multiple lines for the purpose of the fund transfer.

  15. Option to schedule the fund transfer for execution now or at a future date, or set as recurring payment. The NETinfo DBP has an embedded payment scheduler; if cannot be provided by Bank, this can be used for the scheduling of future payments and standing order.

  16. Option to attach documents on the Fund Transfer (maximum file size, eligible file types and maximum number of files configurable). The bank must provide the necessary interfaces for retrieving the document, the relevant document repository and anti-virus scanning.

  17. Option to enforce document attachment above a transfer amount threshold

Step 1: Input Page

The figure below (UI/UX - Step 1: Input - Once Off Beneficiary & UI/UX - Step 1: Input - Saved Beneficiary), presents the ‘Transfer In Rubles’ Input Page. Referring to this, end-user is prompted to complete the following mandatory information:

  • From Account (i.e. the debit account)

  • Beneficiary Name and Account

  • BIK Code by selecting one of the available Banks connected to the associated network using the Search BIK page

  • VO Code by selecting one from a list of available values

  • Amount to be transferred and associated currency

  • Purpose for the transfer. Multiple lines for purpose can be added (up to 4 lines)

  • Charges paid by: SHA / OUR / BEN

  • Scheduling (i.e. immediate execution, execution at a future date, recurring execution/standing order); in case of future date end-user is prompted to enter the ‘Execution Date’ while if the Repeated option is selected, end-user is prompted to define the parameters of the recurring payment i.e. Start Date, Final Payment Date or Number of Executions and Frequency

For better customer experience, the above information can be auto-filled when:

  • The beneficiary has already been registered and a selection of a ‘From Account' has been made; in this case, end-user selects the ‘Saved’ option in the ‘Beneficiary’ field and subsequently selects one of the registered beneficiaries; the Beneficiary Name, Beneficiary Alias, Beneficiary Address, Country, ‘Beneficiary Account, Bank BIK Code, Beneficiary Bank Details, VO Code, KPP Code, Tax Code, Amount & Currency, Purpose, From Description and To Description are automatically auto-filled as available in the registered beneficiary record.

  • Another Fund Transfer has been made in the past to the same beneficiary (using the ‘Once Off’ option); in this case, end-user selects the ‘Once Off’ option in the ‘Beneficiary’ field and starts typing the name of the beneficiary in the ‘Beneficiary Name field; a list of beneficiaries with names matching the characters entered appears dynamically. End-user selects the beneficiary and all information from the past transaction are loaded (same fields as auto-filled using registered beneficiaries).

End-users can optionally:

  • Enter a Beneficiary Address (available as an option when the ‘Once Off’ selection is made)

  • Enter a KPP Code

  • Enter a Tax Code

  • Register the beneficiary details entered (when the 'Once Off selection is made) by clicking on the associated checkbox

  • Provide a description ('From Description') that will appear in their Online Transaction Status and Transaction History (if returned from the Core System).

  • Provide a description (To Description) that will appear in the beneficiary’s transaction details (if returned from the Core System)

  • Use an agreed negotiated rate if the payment amount is in a currency different from the debit account’s currency (end-user has to enter the associated ‘code’).

  • Upload documents to support the Fund Transfer

Once all compulsory information have been entered, end-user proceeds to the next step by pressing the ‘Submit’ button.

A disclaimer is displayed if configured

Confirm and/or Successful SCA

All validations & checks are successful, and end-user approvals are associated with the transaction.

The following actions are taken by the system:

  • The transaction changes status to ‘Pending for Authorisation’

  • The transaction is registered to the ‘Pending Authorisation’ page

  • The transaction is registered to the ‘Online Transaction Status’ page

  • A ‘Pending for Authorisation’ alert is forwarded to the approvers

  • The ‘Ruble Transfer’ alert is forwarded to the inputter

  • The transaction is logged

Confirm and/or Successful SCA

All validations & checks are successful, and transaction has been scheduled for execution at a future date.

The following actions are taken by the system:

  • The transaction changes status to ‘Pending’

  • The ‘Ruble Transfer’ alert is forwarded to inputter

  • The transaction is registered to the ‘Online Transaction Status’ page

  • The transaction is logged

Confirm and/or Successful SCA

All validations & checks are successful, and transaction has been scheduled as a Recurring Payment (Standing Order).

The following actions are taken by the system:

  • The Standing Order is created and handled by the Standing Orders module thereafter

  • The transaction appears in the ‘Online Transaction Status’ page

  • The ‘New Standing Order’ alert is forwarded to the inputter

  • The transaction is logged

Confirm and/or Successful SCA

All validations & checks are successful, and transaction will be executed immediately.

The following actions are taken by the system:

  • The transaction is submitted to the Core System for further processing

  • The status changes to ‘Submit’ (or ‘Failed’ if something goes wrong) and then upon response from the Core System to ‘Completed’ or ‘Rejected’

  • The transaction is registered to the ‘Online Transaction Status’

  • The relevant Fund Transfer alert is forwarded to the end-user when transaction changes status to Failed or Completed or Rejected

  • The transaction is logged

Step 3: Complete Page

Referring to the screen below (UI/UX - Step 3: Complete), the end-user is displayed with the transaction status and the options to:

  • Export the TX details in PDF

  • View Details

  • Initiate a new transfer

By pressing the ‘Done’ button, end-user is redirected to the Transfers and Payments landing page

A disclaimer is displayed if configured
