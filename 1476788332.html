NETinfo DBP Version 18.1 : Standing Orders (Maintained by Bank)

Created by 1057-Michael Philippos, last modified by 1216-Mappoura Andrea on 13, 11, 2024

End-users can create and maintain their standing orders online. Any type of Fund Transfer (if configured) can be set as standing order by defining the associated scheduling parameters i.e. frequency, start date and end-date/number of executions. The creation/amendment/cancellation of a standing order can be associated (if configured) with SCA enforcement, transaction type limits, end-user approval workflows (i.e. signatories) and alerts (upon creation of a standing order).

Note: The Bank must provide a single interface for the creation, amendment, and cancellation instructions submitted by the NETinfo DBP.

Product Main Features / Business Requirements

  1. Fund Transfers can be scheduled to run as standing order by defining the following parameters: frequency (daily, weekly, monthly, quarterly, half-yearly, yearly) and active period (start date and end date/number of executions).

  2. Any type of Fund Transfer can be configured to run as standing order

  3. Standing Orders can be created (by end-user) either during Fund Transfer submission or through a dedicated page

  4. No matter the method used for creating a Standing Order, end-user can view/amend/cancel it at any time through a dedicated user interface

  5. The creation / amendment / cancellation of Standing Orders is treated as any other type of transaction within the NETinfo DBP. As such, it enjoys the related standard features such as:

    1. SCA enforcement (configurable)

    2. End-user approval workflows (configurable)

    3. Alerts (configurable for creation of standing orders)

    4. Online Monitoring (through the ‘Online Transaction Status’ page) with Cancel option

  6. Standing Orders are retrieved from core-banking (no matter the channel used for their creation) and displayed to end-user through dedicated NETinfo DBP pages/screens; as such, the information displayed varies depending on the Bank’s interfaces

  7. The NETinfo DBP merely submits to the Bank’s backend systems the standing orders parameters for further processing and management; the Bank must provide a single interface for the creation, amendment, and cancellation instructions submitted by the NETinfo DBP.

Activity Diagram

The below figure depicts the end-user workflow for creating, amending and cancelling a Standing Orders through the NETinfo DBP channels. A Standing Order can be created:

  • When a Fund Transfer is initiated (by selecting the ‘Repeated’ option and defining the amount, frequency, start  and end dates)

  • By selecting the ‘Create New’ option in the ‘Standing Orders’ page; end-user is prompted to:

    • Select the Fund Transfer type; the corresponding Fund Transfer Input page appears with the ‘Repeated’ option pre-selected (end-user is not allowed to change to either ‘Future Payment’ or ‘Now’).

    • Fill up the payment details and submit;  the standard associated Fund Transfer flow is followed.

End-user can also view all Standing Orders, as maintained in the Bank’s core banking system, through the ‘Standing Orders’ page. While on this page, end-user can:

  • Filter fetched (from the core-banking system) Standing Orders by Status

  • Select a Standing Order for cancellation

  • Select a Standing Order for amendment

  • Create a new Standing Order

  • Export all displayed Standing Order entries either in EXCEL or PFD.

Note that:

  • the NETinfo DBP submits the requests for creation, amendment and cancellation to the core-banking system for execution; the creation & management of the standing orders (including scheduling of payments) is done by the core-banking system.

  • end-user may view the status of the submitted instructions (e.g. create, amend, cancel STO) through the Online Transaction Status

  • through the ‘Standing Orders’ page, end-user may view all standing orders linked to their accounts, as fetched from core-banking, no matter what the channel (e.g. Branch) of creation was

 About the Activity Diagram: The Activity Diagram depicts the main activities in the workflow: boxes indicate end-user activity, decision boxes indicate system decision points and arrow text in brackets indicate major system activity/actions.

Standing Orders

Given that the STO related information displayed through the end-users UI depend on the interfaces provided by the Bank, the fields presented in pages below are indicative.

The figure below (UI/UX - Standing Orders), presents the ‘Standing Orders’ Page. Using this page, the end-user must first select the required account of interest (or all accounts) and then press the ‘GO’ button. All Standing Orders linked to the account(s) are displayed in the reporting area.

End-user can also:

  • Filter these entries by pressing the filter icon and subsequently selecting an Standing Order status

  • Search through the fetched entries using keywords

  • Export the displayed Standing Orders in PDF or EXCEL format

Each Standing Order record contains the following summary information: Ref. No., TX type, Frequency, Next Payment Date and Amount + Currency; additional information can be displayed using the ‘Expand’ option. In the expanded screen the following information is displayed: Contact Name & Details (not included for ‘Fund Transfer between own accounts’), From Account, To Account, Start Date, Final Payment Date, From Description and To Description or Purpose (whichever applies).

End-user can:

  • amend a Standing Order by pressing the ‘Edit’ icon

  • cancel a Standing Order by pressing the ‘delete’ icon

  • create a new Standing Order by pressing the ‘Add New’ action; once selected, end-user is prompted to select a transaction type through a modal and subsequently is navigated to the corresponding Fund Transfer page.

Edit Standing Order

When a end-user selects to edit an existing standing order, the three-step process is followed. During the first step, the end-user can edit certain information of the standing order. The end-user is then prompted to confirm any changes. Once end-user confirms the changes and any relevant SCA validations are completed successfully, the Standing Order is amended or is moved to pending authorisation if signatures are configured. The amendments can be viewed in the Online Transaction Status page.

Edit Standing Order Details

The figure below (UI/UX - Edit Standing Order Details) is displayed when end-user selects to edit an existing standing order. Referring to this, the end-user is displayed in read only format with the ‘From Account’ account and other fields relating to the type of Fund Transfer involved such as:

  • The ‘To’ Accounts as available in the Standing Order record

  • The Beneficiary Name and Account number

End-user can amend the following parameters of the Standing Order:

  • Amount and Currency

  • Frequency

  • Next Payment Date

  • Final Payment Date

By pressing the SAVE button the end-user is navigated to the Confirm page; if the ‘Cancel’ button is pressed, then changes are ignored and end-user is navigated to the ‘Standing Order’ page.

  • The ‘New Standing Order’ alert is forwarded to the inputter

  • The transaction is logged

Cancel Standing Order

When a end-user selects to cancel an active standing order, the end-user is prompted to confirm the cancellation. Once end-user confirms and any relevant SCA validations are completed successfully, the Standing Order is cancelled or is pending authorisation if signatures are configured. The cancellation can be viewed in the Online Transaction Status page.

Cancel Standing Order Confirmation

As shown in the figure below (UI/UX - Cancel Standing Order Confirmation), end-user is displayed with a header and the STO data. End-user can either confirm the transaction by pressing the ‘Confirm’ button or go back to the input page by pressing the ‘Back’ button.

In case the SCA is configured to be enforced for the ‘Cancel STO’ transaction type, then end-user must follow the associated SCA process before being allowed to confirm (If SCA configuration is enabled for the transfer, a modal appears displaying the information required depending on the method of SCA selected.)

In case end-user confirms the cancellation, then the following major system actions take place:

  • The cancel transaction is registered in the Online Transaction status as:

    • ‘Completed’ if no further approvals are required or

    • ‘Pending for Authorisation’ if an approval is required (either through Access rights or Signatures)

  • If no approvals are required, then the STO cancellation takes immediate effect; if approval is required, then the associated approval workflow is followed.

  • The complete page is displayed
