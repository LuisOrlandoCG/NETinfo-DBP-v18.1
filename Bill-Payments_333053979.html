NETinfo DBP Version 18.1 : Bill Payments

Created by Fivi Konstantinou, last modified by 1216-Mappoura Andrea on 13, 11, 2024

End-users can pay their bills (e.g. Utility Bills, Telephone Bills, Taxes, Tuition Fees) online. The bill payment form presented to them can be configured by the Bank (through the NETinfo DBP Admin console) as per the Service Provider’s requirements. The amount due (i.e. bill presentment) can also be displayed to end-users if supported by the Bank.

To facilitate end-users in the bill payment process, several features are provided such as: wizard-based navigation to the bill payment steps, various payment options (e.g. pay all amount due or part), availability of sample bills (uploaded through the NETinfo DBP Admin console) and option to save a bill payment as template for future re-use.

The bill payment instruction is submitted (to the Core System) for execution in real time or at a future date.

Product Main Features / Business Requirements

  1. The bill payment form (used for the payment of a ‘Service’ provided by a ‘Service Provider’) is configured through the NETinfo DBP Admin console

  2. When large number of Service Providers are supported by Bank, capability exists to categorise them to 'Service Provider Categories' (configuration)

  3. Wizard-based navigation of end-user through the bill payment steps

  4. Bill presentment is supported

  5. Uploading of sample bills (for end-user to view as/if required) supported through the NETinfo Admin Console

  6. Bill payment templates (saved by end-user for later re-use) are supported

  7. Eligible debit accounts are:

    1. Accounts that fall under a Bank Product that has been configured in NETinfo DBP to allow debits (from its accounts) for any Fund Transfer type.

    2. Accounts that end-user has Input privileges (for the particular transaction type) as defined in User Profile.

    3. Accounts under the Customer selected by end-user from the ‘Owner’ field.

  8. The eligible currencies involved in a transfer and a cross currency transfer are configurable

  9. Alerts are supported (configurable).

  10. Calculation of charges is supported (configurable).

  11. Signatures are supported (configurable).

  12. SCA functionality is supported (configurable).

  13. Daily limits at customer, account, user and user-account level can be enforced (configurable).

  14. Minimum and Maximum Transaction Limits can be enforced (configurable).

  15. Option to schedule the payment for execution now or at a future date. The NETinfo DBP has an embedded payment scheduler; if cannot be provided by Bank, this can be used for the scheduling of future payments.

Activity Diagram

The below figure presents the user Activity Diagram for the Bill Payments function. Referring to this, the end-user first selects the required Service by selecting:

  • the Service Provider Category

  • the Service Provider

  • the Service

In case the selected Service supports ‘Bill Presentment’, then the bill payment details are filled in a two-step process as follows:

  • Step 1: End-user defines the debit account and the bill reference no(s) – configurable; system fetches the ‘Debt Amount’ based on the reference no(s) 

  • Step 2: End-user defines payment related parameters i.e. either pays the full amount or part of it, pays now or later

In case the selected Service does not support ‘Bill Presentment’, then end-user completes all payment details in a single step.

For convenience, the end-user has also the option to use a pre-saved ‘Bill Payment Template’, in which case certain steps are bypassed and the bill payment input page is prefilled with the following information:

  • Service Provider

  • Service

No matter the path taken by end-user for completing the payment transaction details, upon submitting the system:

  • Performs field validations

  • Checks whether approval workflows are linked to the payment. In case of approval workflows, system only performs Transaction Type limit check; alternatively, system also performs credit check and full limits checks

The payment details are displayed in the ‘Confirm’ page; upon end-user confirmation, system checks whether SCA is required (in such case end-user follows the SCA process), conducts again the above checks and:

  • Submits payment instruction to Core System for execution in case of no approval workflows

  • Notifies approvers in case of approval workflow

  • Registers transaction in the ‘Online Transaction Status’ and system log

  • Forwards the ‘Bill payments’ alert

About the Activity Diagram: The Activity Diagram depicts the main activities in the workflow: boxes indicate user activity, decision boxes indicate system decision points and arrow text in brackets indicate major system activity/actions.

Bill Payments

To complete a Bill Payment, the end-user is navigated through a 5 step process using a form wizard. The end-user is prompted to enter data or make a selection in each step and proceed to the next step until completion.

Using the wizard, the end-user is prompted to select:

  • between Template or New Payment options

  • a Service Provider Category

  • a Service Provider

  • a Service offered by the service provider

  • any relevant payment details

Once the 5 step process is completed, the end-user confirms the details entered and completes the process.

The figure below (UI/UX - Bill Payments Main Page), presents the ‘Bill Payments’ Page. Using this page, the end-user can either:

  • Select an existing bill payment template from the available list, in which case s/he is redirected to the Final Input Payment page. This list displays the three most recently used templates; in case end-user wants to view all available templates then ‘Show All’ option can be selected.

Final Input Payment

The figure below (UI/UX - Final Input Payment - Bill Presentment exists), is displayed ONLY when the ‘Bill Presentment’ service is enabled, AND:

  • EITHER the end-user has successfully submitted the ‘Define Reference and DR account’ page OR

  • The payment has been initiated from a Bill Payment Template.

Referring to this, then end-user:

  • Views (read only) the values entered for Service and Reference Number (+ any other fields as configured through the Admin console)

  • Views the default ‘From Account' as selected from previous page with the option to change it as required

  • Views the Debt Amount (fetched from Core System based on the Reference Number)

  • Selects whether the ‘Debt Amount’ will be paid in pull or a specific amount will be paid; in case the ‘Specify’ option is selected, then the ‘Debt Amount’ field becomes editable (if the ‘Full’ option is selected, the ‘Debt Amount’ field is not editable)

  • Selects whether the payment will be executed NOW or at a LATER date; if the ‘Later’ option is selected, then end-user must define the future execution date

The figure below (UI/UX - Bill Payments Step 5: Final Payment Details - Bill Presentment does not exist), presents the final input page when bill presentment is NOT available. It is triggered by either:

  • A payment initiated from a ‘Bill Payment Template’ (i.e. end-user has selected a bill payment template in the ‘Bill Payments’ main page), OR

  • After the selection of an SP from the ‘Select Service Provider’ page

Referring this, then end-user will:

  • Select the ‘From Account' (default as selected from previous page)

  • Enter the amount to be paid in field ‘Amount’

  • Select whether the payment will be executed NOW or at a LATER date; if the ‘Later’ option is selected, then end-user must define the future execution date

  • Complete any other fields as configured through the Admin console

In both pages, the end-user can opt to navigate to the previous page by selecting the ‘Back’ button or submit the payment for execution by pressing the ‘Submit’ button.

Confirm and/or Successful SCA

All validations & checks are successful, and end-user approvals are associated with the transaction.

The following actions are taken by the system:

  • The transaction changes status to ‘Pending for Authorisation’

  • The transaction is registered to the ‘Pending Authorisation’ page

  • The transaction is registered to the ‘Online Transaction Status’ page

  • A ‘Pending for Authorisation’ alert is forwarded to the approvers

  • The ‘Bill Payment’ alert is forwarded to the inputter

  • The transaction is logged

  • End-user is redirected to Complete

Confirm and/or Successful SCA

All validations & checks are successful, and transaction has been scheduled for execution at a future date.

The following actions are taken by the system:

  • The transaction changes status to ‘Pending’

  • The ‘Bill Payment’ alert is forwarded to inputter

  • The transaction is registered to the ‘Online Transaction Status’ page

  • The transaction is logged

  • End-user is redirected to Complete

Confirm and/or Successful SCA

All validations & checks are successful, and transaction will be executed immediately.

The following actions are taken by the system:

  • The transaction is submitted to the Core System for further processing

  • The status changes to ‘Submit’ (or ‘Failed’ if something goes wrong) and then upon response from the Core System to ‘Completed’ or ‘Rejected’

  • The transaction is registered to the ‘Online Transaction Status’

  • The relevant ‘Bill Payment’ alert is forwarded to the end-user when transaction changes status to Failed or Completed or Rejected

  • The transaction is logged

  • End-user is redirected to Complete

Text

Read Only

Status of the transaction & NETinfo DBP Reference Number for the transaction and UTN (Unique Transaction Number) retrieved from Core System (if applicable).

Data:

  • Transaction Status: [display one of the statuses below]

  • Reference Number [display number] | UTN [display number]

The possible Transaction Status are:

  • Pending: Transaction is pending submission at future execution date

  • Pending for Authorisation: Transaction is pending end-user authorization

  • Failed: Transaction has failed to be submitted to Core System for execution

  • Submitted: This is a transit state - transaction has been submitted to Core System but no valid response has been received

  • Pending Execution: Transaction has been submitted and Core System have responded; however processing by Core System have not completed (e.g. could be some kind of manual processing)

  • Completed: Transaction has been processed successfully by bank’s Core System

  • Rejected: Transaction has been rejected by bank’s Core System
