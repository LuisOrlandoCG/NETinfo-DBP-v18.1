NETinfo DBP Version 18.1 : Fund Transfer Between Own Accounts

Created by NETInfo Plc, last modified by 1216-Mappoura Andrea on 08, 11, 2024

Fund transfer between own accounts enables the transfer of funds between the end-user’s connected accounts. The payment instruction can be submitted (to the Core System) for execution in real time or at a future date or as a standing order. As with any other type of Fund Transfer, Transfer between Own Accounts conforms to the three step process (i.e. submit, confirm, complete) and certain configuration needs to be done at the Administrator console for its use.

Product Main Features / Business Requirements

  1. Eligible debit accounts are:

    1. Accounts that fall under a Bank Product that has been configured in NETinfo DBP to allow debits (from its accounts) for the particular transaction type

    2. Accounts that end-user has Input privileges (for the particular transaction type) as defined in User Profile.

    3. Accounts under the Customer selected by end-user from the ‘Owner’ field.

    4. Accounts of allowed currencies as defined in the Transaction Type Currencies

  2. Eligible credit accounts are:

    1. Accounts that fall under a Bank Product that has been configured in NETinfo DBP to allow credits (from its accounts) for the particular transaction type

    2. Accounts that end-user has Input privileges (for the particular transaction type) as defined in User Profile.

    3. Accounts under the Customer selected by end-user from the ‘Owner’ field

    4. Accounts of allowed currencies as defined in the Transaction Type Currencies

  3. Cross currency transfers are supported with the option to use a negotiated rate. In a cross currency transfer, the currencies of the Debit Account and Transfer Amount differ.

  4. The Transfer Amount’s eligible currencies in same-currency (i.e. when Transfer Amount currency is the same as the Debit Account currency) and cross currency (i.e. when Transfer Amount currency is different than the Debit Account currency) transfers are configurable through the Transaction Type Currency Admin page; the Transfer Amount currency is validated against the eligible currencies at Submit.

  5. Signatures are supported (configurable).

  6. Calculation of charges is supported (configurable)

  7. SCA functionality is supported (configurable).

  8. Daily limits at customer, account, user and user-account level can be enforced (configurable).

  9. Minimum and Maximum Transaction Limits can be enforced (configurable).

  10. Option to schedule the fund transfer for execution now or at a future date, or set as recurring payment. The NETinfo DBP has an embedded payment scheduler; if cannot be provided by Bank, this can be used for the scheduling of future payments and standing order.

Step 1: Input

The figures below (UI/UX - Immediate Execution of Payment, UI/UX - Future Payment & UI/UX - Recurring Payment), presents the ‘Fund Transfer between Own Accounts’ input page. Referring to this, end-user is prompted to complete the following mandatory information:

  • From account (i.e. the debit account)

  • To account (i.e. the credit account)

  • Amount to be transferred and associated currency

  • Scheduling (i.e. immediate execution, execution at a future date, recurring execution/standing order); in case of future date, end-user is prompted to enter the ‘Execution Date’ ('UI/UX Future Payment) while if the Repeated option is selected, end-user is prompted to define the parameters of the recurring payment i.e. Start Date, Final Payment Date or Number of Executions and Frequency (refer to ‘UI/UX Recurring Payment’)

End-user can also add a description where it will appear on the transaction history of the ‘Pay From’ Account and a description that will appear in the transaction history of the ‘Pay To’ Account.

In case the transfer amount is in a currency different from the debit account’s currency, then the option to use a negotiated rate is also available (end-user has to enter the associated ‘code’).

End-user proceeds to the next step by pressing the ‘Submit’ button.

Confirm and/or Successful SCA

All validations & checks are successful, and end-user approvals are associated with the transaction.

The following actions are taken by the system:

  • The transaction changes status to ‘Pending for Authorisation’

  • The transaction is registered to the ‘Pending Authorisation’ page

  • The transaction is registered to the ‘Online Transaction Status’ page

  • A ‘Pending for Authorisation’ alert is forwarded to the approvers

  • The ‘Transfer Between Own Accounts’ alert is forwarded to the inputter

  • The transaction is logged

Confirm and/or Successful SCA

All validations & checks are successful, and transaction has been scheduled for execution at a future date.

The following actions are taken by the system:

  • The transaction changes status to ‘Pending’

  • The ‘Transfer Between Own Accounts’ alert is forwarded to inputter

  • The transaction is registered to the ‘Online Transaction Status’ page

  • The transaction is logged

Confirm and/or Successful SCA

All validations & checks are successful, and transaction has been scheduled as a Recurring Payment (Standing Order).

The following actions are taken by the system:

  • The Standing Order is created and handled by the Standing Orders module thereafter

  • The transaction appears in the ‘Online Transaction Status’ page

  • The ‘New Standing Order’ alert is forwarded to the inputter

  • The transaction is logged

Confirm and/or Successful SCA

All validations & checks are successful, and transaction will be executed immediately.

The following actions are taken by the system:

  • The transaction is submitted to the Core System for further processing

  • The status changes to ‘Submit’ (or ‘Failed’ if something goes wrong) and then upon response from the Core System to ‘Completed’ or ‘Rejected’

  • The transaction is registered to the ‘Online Transaction Status’

  • The relevant Fund Transfer alert is forwarded to the end-user when transaction changes status to Failed or Completed or Rejected

  • The transaction is logged

The possible Transaction Status are:

  • Failed: Transaction has failed to be submitted to Core System for execution

  • Submitted: This is a transit state - transaction has been submitted to Core System but no valid response has been received

  • Pending Execution: Transaction has been submitted and Core System has responded; however processing by Core System has not completed (e.g. could be some kind of manual processing)

  • Completed: Transaction has been processed successfully by bank’s Core System

  • Rejected: Transaction has been rejected by Core System
