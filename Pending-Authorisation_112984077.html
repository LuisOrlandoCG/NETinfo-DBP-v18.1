NETinfo DBP Version 18.1 : Pending Authorisation

Created by Fivi Konstantinou, last modified by 1057-Michael Philippos on 19, 11, 2024

When a transaction linked to an approval workflow is initiated from any of the channels connected to the NETinfo DBP, the assigned signatories receive an alert notifying them of the event. Signatories can subsequently approve or decline the transaction as required.

Product Main Features / Business Requirements

  1. End-user approval workflows are linked to a transaction type and debit account

  2. End-user approval workflows may also be linked to customer, in which case approval workflows apply for all debit accounts linked to the customer

  3. A checker (i.e. end-user that has ‘Approve’ privileges on a specific Transaction Type and Bank Customer/CIF) may approve transactions linked to a Bank Customer/CIF (not account) as applicable

  4. End-user approval workflows, signatories and associated alert are configurable through the NETinfo DBP

  5. Assigned signatories are notified via an alert (the ‘Pending for Authorisation’ alert) when a transaction is pending on them for authorisation

  6. Assigned signatories may view the transactions pending on them for authorisation through the ‘Online Transaction Status’ and the ‘Pending Authorisation’ pages

  7. End-user can enable/disable the ‘Pending for Authorisation’ alert, and define the associated delivery channel, through their user interface

  8. Authorised signatories can approve or decline transactions (one by one, or all transactions at once) through their end-user interface

  9. Fund transfers with a debit-credit account combination encountered for the first time are highlighted.

  10. Charges are re-calculated at the pending authorisation flow

  11. The maximum number of end-user approvals at a time may be limited through configuration

  12. Transactions pending authorisation can be exported in Excel or PDF.

Activity Diagram

The figure below presents the activity diagram of the pending authorisations function. Referring to this, an approver opens the ‘Pending Authorisation’ page and selects the Account of interest or ‘Customer based transactions’ option as applicable; system fetches all related transactions that have been initiated from the NETinfo DBP and require an approval from the end-user.

End-user can further filter the transactions fetched through keyword search or using:

  • Amount range

  • Period

  • Transaction Type

  • Currency

End-user can perform the following actions on transactions displayed:

  • Approve a single or all transactions shown at once (maximum number of approvals limited by the ‘maxAllowedUserApprovals’ setting)

  • Decline a single or all transactions shown at once

If end-user is the last approver, then the NETinfo DBP conducts all required transaction checks such as Credit Check and Limit Checks.

End-user can also export the displayed transactions to PDF or EXCEL file.

About the Activity Diagram: The Activity Diagram depicts the main activities in the workflow: boxes indicate user activity, decision boxes indicate system decision points and arrow text in brackets indicate major system activity/actions.

Pending Authorisation Input Page

The figure below (UI/UX - Pending Authorisation), presents the ‘Pending authorisation’ page. Referring to this, when an approver opens the ‘Pending Authorisation’ page, they first select the account of interest or ‘Customer Based Transactions’ option as required and subsequently press the GO button; the system fetches all related transactions that have been initiated from the NETinfo DBP and require an approval from the end-user (by default, transactions for approval on all accounts are displayed).

End-user can further filter the transactions fetched through keyword search or using the filter option which enables filtering by:

  • Amount range (From Amount’ and ‘To Amount’ fields)

  • Currency

  • Period ('From' and ‘To’ dates)

  • Transaction Type

The filtering option is executed when the ‘Apply’ button is pressed; filtering options can be cleared by pressing the ‘Clear’ button

End-user can view the following header information for each transaction:

  • Transaction Reference Number: NETinfo DBP unique reference number

  • Initiation date: The date of submission/creation.

  • Transaction Type

  • Amount & Currency

By clicking on the expand option, additional transaction-related details appear as below:

  • Reference Number (this is the Transaction Reference Number)

  • Transaction Description: All mandatory and major optional fields completed by the end-user (i.e. fields with values) when submitting the transaction are displayed; typical information that might be displayed are shown below:

    • From Account

    • To Account

    • To (Own Card/Other NETinfo Card)

    • Card Number

    • Cardholder Name

    • Beneficiary Name

    • Beneficiary Bank

    • Debit Amount/Currency

    • Execution Date

    • Charges Paid By

    • Start Date

    • Final Payment Date

    • Frequency

    • From Description

    • To Description

    • Attachment

    • Service Category

    • Service

    • Dynamic Page fields (those configured for display)

  • Additional information, such as the below, are displayed as applicable:

    1. Negotiation Rate: [Negotiated rate] to be applied if applicable

    2. Charges: Charges as applicable when end-user has submitted the transaction

    3. Exchange Rate: Actual Exchange Currency rate to be used [Retrieved from Core System when end-user has submitted the transaction]

    4. Debit Amount: Actual amount to be debited from end-user’s account [Retrieved from Core System when end-user has submitted the transaction)

  • Approval details:

    • Name of end-user who created the transaction

    • Approvers' names

    • Remaining Signatures

End-user may select transactions for Approval/Decline either:

  • One at a time, using the associate Approve/Decline buttons, or

  • All at once, using the ‘Approve All' or 'Decline All’ buttons

In any case, the processing of end-user’s selections take effect upon pressing the ‘Submit’ button (this button is displayed only after selection/s have been made), in which case end-user is navigated to the ‘Confirm’ page; alternatively, end-user may cancel their selections by pressing the ‘Cancel’ button (this button is displayed only after selection/s have been made)

End-user may also export the displayed transactions to PDF or EXCEL file.
