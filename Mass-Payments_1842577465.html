NETinfo DBP Version 18.1 : Mass Payments

Created by 1057-Michael Philippos, last modified on 14, 11, 2024

End-users can upload payment files (e.g. payroll, supplier payments) in:

  • Character delimited TXT file formats, the structure of which is configured through the administration console, or

  • SEPA XML format in accordance with the ISO 20022 international standard and in compliance with the Regulation (EU) 260/2012 (SEPA); this is not configurable.

The business rules (e.g. account validations, access rights, limits, credit check) that apply for standard fund transfers are applied before submitting the file (to the Core System) for payment execution. For non-SEPA XML file formats, additional validations as defined by the bank administrator in the Admin Console apply.

Product Main Features / Business Requirements

  1. Mass payment files may include any type of fund transfer

  2. Two file formats supported: delimited TXT and SEPA XML in accordance with the ISO 20022 international standard and in compliance with the Regulation (EU) 260/2012 (SEPA)

  3. Mass payment files in TXT format are validated against the selected (by end-user) mass file template (configured by admin-user through the ‘File Template’ field in NETinfo Admin Console)

  4. Mass payment file may be submitted for processing ‘now’ or at a future date, as selected by end-user (The NETinfo DBP has an embedded payment scheduler; if cannot be provided by Bank, this can be used for the scheduling of future payments).

  5. Mass payment file in TXT format may include headers and/or footers as defined in mass file template

  6. Single Debit Multiple Credit capability (debit account is selected by end-user from list of eligible accounts)

  7. Mass payment description, as defined by end-user (through the ‘Description’ field), appears in:

    • Mass Transaction Status

    • Mass Authorisation

    • Transaction History (note: this depends on interfaces)

  8. The following validations are taking place:

    1. File Validations: Correct file format & structure (taking place when uploading mass payment file)

    2. Record Validations: Validations against mass payment template when in TXT format; field validations when file uploaded is in XML format (taking place when uploading mass payment file)

    3. Mass Payment Validations: Standard checks relating to the ‘Mass Payment’ transaction conducted during submission e.g. Mass payment Transaction Type Limits, Daily Limits, Credit Check, input ‘Mass Payments’ page field validations etc.

    4. Fund Transfer Validations: Standard checks against each fund transfer instruction within the mass payment file e.g. Fund Transfer field validations, Fund Transfer Transaction Type Limits (conducted when submitting the mass payment file)

  9. Exception messages relating to ‘Record Validations’ and ‘Fund Transfer Validations’ may be viewed at payment instruction (record) level

  10. Fund transfer instructions / records in error may be skipped (configurable)

  11. Mass payments are supported by two dedicated alerts: One for Inputter ('Mass Payment') and the other for the Approver ('Mass Authorisation')

  12. Enable / disable configuration for cross-currency payments within mass payments file; if enabled, constrains on currencies as configured for the various types of fund transfers apply (configurable).

  13. Signatures are supported at ‘Mass Payment’ transaction type level (configurable).

  14. Calculation of charges is supported (configurable)

  15. SCA functionality is supported (configurable).

  16. Daily Limits at customer, account, user and user-account level can be enforced (configurable).

  17. Minimum and maximum ‘Mass Payment’ Transaction Type Limits can be enforced (configurable).

  18. Existing Transaction Type Limits on fund transfers are enforced

  19. Submission of fund transfer instruction for execution take place asynchronously

  20. The maximum number of fund transfer instructions per mass payment file is 60.000

  21. Mass Payment submissions appear in ‘User Reports’: Daily Activity Report, Online Transactions Report, Exceptions Report and Fund Transfer Totals Report

  22. Admin-user actions on mass payment configurations appear in Audit and Exceptions Reports

Activity Diagram

The figure below presents the end-user activity diagram (workflow) for mass payments. Referring to this, when end-user opens the ‘Mass Payments’ web page, they first have to fill certain basic input parameters relating to the mass payment transaction (e.g. select ‘From Account’ account, ‘File Delimiter’ and ‘File template’ and define file ‘Description’).

When end-user selects to upload the mass payment file, certain ‘File Validation’ checks are performed by the system (e.g. file has not been uploaded before, file type is TXT or XML etc.); if any validation fails then error messages are displayed and further processing is paused. If there are no ‘File Validation’ exceptions, then system uploads file and performs ‘Record Validations’ (these are mainly validations based on the configurations in associated mass file template done in the Admin Console - conducted when file is in TXT format or field validations when file is in XML format); error messages are displayed at record level.

In case of errors and the ‘massIgnoreErrors’ flag is not set, then the process terminates here; if the flag ‘massIngoreErrors’ is set, then end-user has the option to skip records in error and proceed with only valid records.

Upon pressing the ‘Submit’ (in case of no errors) or ‘Skip and Submit’ (in case of errors and the flag ‘massIgnoreErrors’ is set), the system performs the ‘Mass Payment Validations’ (i.e. Mass Payment TX Limit check, Daily Limit checks and Fund Availability check: note that  Daily Limits & Fund Availability checks are performed only if the transaction will be executed immediately); if any of these checks fail, then the process is terminated.

If ‘Mass Payment Validations’ are successful, then the ‘Fund Transfer (FT) validations’ are performed (e.g. check that the beneficiary account exists if FT between own accounts or within bank, IBAN/BIC/BIK/Clearing code is valid if/as required etc.). Possible errors are displayed at record level. In case of errors and the ‘massIgnoreErrors’ flag is not set, then the process terminates here; if the flag ‘massIngoreErrors’ is set, then end-user has the option to skip records in error and proceed with only valid records.

Upon successful submission, end-user is displayed with the ‘Confirm’ page; end-user views data, follows SCA workflow if/as required and valid records are submitted asynchronously for further processing; workflow depends on execution date and existence of approvers (same workflow applies as when any FT is submitted)

About the Activity Diagram: The Activity Diagram depicts the main activities in the workflow: boxes indicate end-user activity, decision boxes indicate system decision points and arrow text in brackets indicate major system activity/actions.

Mass Payments: Input Page

The three figures below (UI/UX - Mass payments before upload, UI/UX - Immediate execution of payment with no errors & UI/UX - Mass payment with skip and submit option), present:

  • The ‘Mass Payments’ input page before the payment file is uploaded

  • The ‘Mass Payments’ input page, after uploading a payment file with correct data

  • The ‘Mass Payments’ input page, after uploading a payment file and exceptions (in ‘Record Type’ and ‘Fund Transfer’ validations) exist

Referring to the first screen, end-user is prompted to complete the following mandatory information:

  • From Account (i.e. the debit account): end-user selects one of the eligible accounts that appear in the drop down list

  • File Template: end-user selects either one of the existing Mass File Templates (as defined through Mass Payments Configuration) or the XML file option

  • File Delimiter: enabled only when an existing Mass File Template has been selected (i.e. for TXT format files); end-user selects the delimiter character that applies in the payment file

  • Description: end-user enters a description for the Mass Payment

  • Scheduling: for immediate execution, the ‘NOW’ option is selected; in case of future date end-user is prompted to enter the ‘Execution Date’ ('UI/UX Future Payment)

  • Flag to ignore first and last line(s): enabled only when an existing Mass File Template has been selected (i.e. for TXT format files); if selected, the first and last lines as defined in the Mass File Template are ignored

End-user uploads the payment file using the ‘Upload' button; any exceptions arising from ‘File Validations’ appear on the top of the page. In case of no exceptions (refer to the second figure), the records in the mass payment file are loaded and appear in the ‘File Records’ table (expand/collapse option is available) along with a summary of the totals at the bottom of the page (the number of payment instructions and total equivalent amount is shown; if expanded, the total amount and number of transactions per currency is displayed). In case of exceptions from ‘Record Type’ validations (refer to the third figure), only payment transactions in error appear along with associated exception messages (the totals are not displayed). If configured, the end-user is given the option to ‘Skip & Submit’ or cancel’ the process by pressing the ‘Clear’ button.

If end-user proceeds by pressing the ‘Submit’ (or ‘Skip & Submit’) button, system performs the ‘Mass Payment’ and ‘Fund Transfer’ validations; in case of exceptions in ‘Mass Payment’ validations, exception messages are displayed on the top of the page, and the process is terminated. In case of exceptions in ‘Fund Transfer’ validations, only payment transactions in error appear along with associated exception messages (the totals are not displayed). If configured, the end-user is given the option to ‘Skip & Submit’ (in which case end-user is navigated to the ‘Confirm’ page) or cancel the operation by pressing the ‘Clear’ button.

A disclaimer may be displayed if configured

All validations & checks are successful, and end-user approvals are associated with the transaction.

The following actions are taken by the system:

  • Τhe Mass Payment Transaction Status is set to ‘Pending for Authorisation’ (or ‘Failed’)

  • The transaction is recorded in the ‘Mass Authorisation’ and ‘Mass Transaction Status’ pages

  • The ‘Mass Authorisation’ alert is forwarded to approvers

  • The ‘Mass Payment’ alert is forwarded to the inputter

  • Store the File Name and associated Customer linked to the file; this will be used to stop end-user from uploading the same file next time

  • Log the event

  • Display the ‘Complete Page’

Confirm and/or Successful SCA

All validations & checks are successful, and transaction has been scheduled for execution now or at a future date - no approvals are associated with the transaction.

The following actions are taken by the system:

  • Τhe Mass Payment Transaction Status is set to ‘Pending’ (or ‘Failed’)

    • IF execution is NOW THEN it will be executed based on the ‘Pending Batches’ timer

    • IF execution is for a future date THEN it will be scheduled for execution on that date (if the NETinfo scheduler is used)

  • The transaction is recorded in the ‘Mass Transaction Status’ page

  • The ‘Mass Payment’ alert is forwarded to the inputter

  • Store the File Name and associated Customer linked to the file; this will be used to stop end-user from uploading the same file next time

  • Log the event

  • Display the ‘Complete Page’
