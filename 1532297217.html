NETinfo DBP Version 18.1 : SCA (SMS OTP and Push Notification)

Created by 1192-Eliades Soteris, last modified by 1216-Mappoura Andrea on 13, 11, 2024

Strong Customer Authentication (SCA) is a requirement for PSD2 compliance. PSD2 (Revised Payment Services Directive) is a set of laws and regulations for payment services in the European Union (EU) and the European Economic Area (EEA).

SCA demands the use of two or more elements categorised as knowledge (something only the user knows), possession (something only the user possesses) and inherence (something the user is) that are independent, in that the breach of one does not compromise the reliability of the others, and is designed in such a way as to protect the confidentiality of the authentication data.

The NETinfo SCA Module is fully compliant with PSD2, and provides the below methods:

  1. SMS OTP

  2. Push Notification

Scenario 1: SMS Tan

The figure below (UI/UX - Authentication - SMS OTP), presents the ‘Two factor authentication’ modal. Referring to this, the end-user has to authenticate themselves using the SCA method configured by the admin.

This scenario occurs when:

  • SMS Tan option has been selected as the SCA Method in User’s Profile (User Management page/Configuration tab)

If SCA is enabled, an SMS OTP is sent on the user’s registered mobile phone number and must be inputted on the screen in order to complete the action.

Scenario 2: Push Notification

The figure below (UI/UX - Authentication - Push), presents the ‘Two factor authentication’ modal. Referring to this, the end-user has to authenticate themselves using the SCA method configured by the admin.

This scenario occurs when:

  • Push option has been selected as the SCA Method in User’s Profile (User Management page/Configuration tab)

If SCA is enabled, a Push notification is received on end-user’s mobile app. The end-user can then accept or decline the transaction, the user will be redirected to the appropriate page.

Screen Actions & Event Handling

Object

Event

Actions

Confirm

User selects Confirm (on Web)

AND

Push Notifications are enabled.

Push Notification is sent if enabled.

A modal appears on the page and the following actions will take place:

  • Information for the customer appears

  • A push notification is sent to the registered device

  • PIN/Biometrics are required for authentication to login to the mobile app. For the authentication of PIN/Biometrics, internet connection is required on mobile.

  • Push Notification is opened and user can Approve or Reject the transaction within a specific time frame. The time frame is configurable through administration console, under System Properties: SCA push time out.

  • Details of transaction are displayed on push notification (See Push Notification Details section below)

Close

End-user selects to close the modal.

Modal closes.

Push Notification - <Close>

User selects the 'X’ icon.

The request is still active, the user can access the request by logging in the mobile app again.

Push Expires

The push has expired

Display Error Message: ‘Time has expired’.

Resend Push

User selects Resend Push.

A Push Notification is regenerated and forwarded to the user.

Cancel

User selects Cancel.

Modal closes and user returns to the Confirm page.

Display Error Message: ‘OTP code has expired.’

If user has the push notification open and selects Cancel from the Confirm page on the Web, they will not be able to approve/reject the push notification and a message will be displayed on the mobile device ‘Invalid code.’

Reject

User selects Reject.

Dialog closes and transaction is updated accordingly. The request is rejected. Display Error Message: 'Authorisation rejected'.

Upon selecting 'X', the modal closes and the user views the Confirm page.

Approve

User selects Approve.

Dialog closes and transaction is updated accordingly. The request is approved.

User is redirected to the Complete page.
