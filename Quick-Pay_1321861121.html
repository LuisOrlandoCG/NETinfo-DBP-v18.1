NETinfo DBP Version 18.1 : Quick Pay

Created by Fivi Konstantinou, last modified by 1246-Spyrou Marios on 04, 10, 2024

End-users can register and use the quick pay service for easy payments to/from other registered for the service end-users. When making a Quick Pay payment, the end-user may select the beneficiary by entering their mobile number or selecting them from their phone’s contact list.

NOTE: he Mobile app pages are implemented within the scope of the Project; the NETinfo DBP product only provides the required APIs

Product Main Features / Business Requirements

  1. End-user can select to register to or unregister from the quick pay service; only registered end-users can use the service

  2. For the Quick Pay service, end-users will be linked to the mobile number registered in their User Profile (the relationship between end-user and mobile number is unique)

  3. When registering, end-users:

    1. Must select the default account to be used in quick pay (this will be used for debit and credit purposes) from a list of eligible accounts

    2. May read the associated Terms and Conditions (these are not configurable)

  4. Eligible debit accounts are:

    1. Accounts that fall under a Bank Product that has been configured in NETinfo DBP to allow debits (from its accounts) for the particular transaction type

    2. Accounts that end-user has Input privileges (for the particular transaction type) as defined in User Profile.

    3. Accounts under the Customer selected by end-user from the ‘Owner’ field.

  5. When making a Quick Pay payment, end-user:

    1. May use the existing default debit account (as selected in the ‘Quick Pay Registration’ page) or select a new account (from a list of eligible accounts) for the specific transaction

    2. Must enter the beneficiary by either entering the associated mobile number or selection through their phone’s contact list

    3. Must enter the amount to be paid

    4. May add a description of the payment

  6. Upon entering the beneficiary’s mobile number (either manually or through contact list), system should validate it and:

    1. Return the Account Number and Beneficiary’s Name & Surname (as stored in the associated User Profile) in masked format if the mobile number exists in a User Profile, or

    2. Return an exception message if the mobile number is not found

  7. The Quick Pay payment transaction conforms to the three steps process: Input, Confirm, Complete

  8. Only base currency payments are allowed

  9. SCA can be enforced (configurable)

  10. Daily limits at customer, account, user and user-account level can be enforced (configurable)

  11. Minimum and Maximum Transaction Limits can be enforced (configurable)

  12. The following features (that apply for Fund Transfers) are not applicable for the Quick Pay service:

    1. Option to schedule the fund transfer at a future date, or set as standing order

    2. Disclaimers

    3. Signatures

    4. Calculation of charges

Activity Diagram

Figure below depicts the end-user workflows relating to the ‘Quick Pay’ function. Referring to this and given that the ‘Quick Pay’ function is only available for registered (to the service) end-users, the first thing that needs to be done is registration. The associated workflow for registration is displayed in the left hand side of the diagram. Basically, end-user needs to:

  • Toggle the ‘Quick Pay’ option in Settings, and

  • Select the account that will be used by default for debiting/crediting initiated by the Quick Pay function (this will be their default ‘Quick Pay’ account); the default account may be changed at any time

The associated T&Cs may be displayed at any time

Upon registration, the end-user is ‘known’ to the Quick Pay function by their mobile number. A registered end-user can thereafter select to unregister (by toggling the Quick Pay option in Settings - see right hand side flow in diagram) or make a 'Quick Pay' payment (refer to the workflow in the centre of the diagram) by:

  • Optionally selecting a debit account different than the ‘Quick Pay’ account

  • Selecting a beneficiary using one of the following methods:

    • Manual entry of mobile number of beneficiary

    • Selection of beneficiary from their phone’s contact list

    • Selecting a beneficiary from the ‘Recent Beneficiary’ list

  • If the beneficiary selected is a (valid) registered ‘Quick Pay’ end-user, then the beneficiary name and account are displayed in masked format (if beneficiary is not found, an exception message appears) and the end-user is prompted to enter the amount to be transferred and (optionally) a description

  • End-User must Submit and subsequently Confirm the transfer of money (money will be transferred from the end-users debit account to the beneficiary’s Quick Pay account)

About the Activity Diagram: The Activity Diagram depicts the main activities in the workflow: boxes indicate end-user activity, decision boxes indicate system decision points and arrow text in brackets indicate major system activity/actions.

Enabling the Quick Pay Service

The figures below, ‘UI/UX Enable Quick Pay Service’ pages, present the end-user registration / un-registration process to the Quick Pay service. Referring to these, end-user:

  • May select to view the Quick Pay’s Terms and Conditions

  • Toggle the ‘Quick Pay’ registration option:

    • if Quick Pay is disabled, then by toggling to ‘Enabled’ end-user:

      • is prompted to confirm their intention of registering to the service, and

      • subsequently, is prompted to select the default Quick Pay account (this will be the default account for all inbound/outbound Quick Pay transfers)

      • may change the default account at any time

    • If Quick Pay is enabled, then by toggling to ‘disabled’ end-user is prompted to confirm action

STEP 1/2: Input

The figures below, ‘UI/UX Quick Pay Transfer’ pages, present the process of initiating a quick pay transfer to a beneficiary that is registered to the quick pay service. Referring to these, end-user is prompted to complete the following mandatory information:

  • Pay from Account (i.e. the debit account); end-user may select a debit account different than the default Quick Pay account

  • Beneficiary Phone Number: This can be defined in three ways:

    • By manually entering the mobile number of the beneficiary

    • By selecting a beneficiary from their phone’s contact list

    • By selecting a beneficiary from the ‘Recent Recipients’ list

No matter the method used for defining the beneficiary, system validates whether the assigned beneficiary is a registered Quick Pay end-user; if not, an exception message appears

Once all compulsory information have been entered, end-user proceeds to the next step by pressing the ‘Next’ button.

STEP 3: Input Amount

Confirm and/or Successful SCA

All validations & checks are successful, and transaction will be executed immediately.

The following actions are taken by the system:

  • The transaction is submitted to the Core System for further processing

  • The status changes to ‘Submit’ (or ‘Failed’ if something goes wrong) and then upon response from the Core System to ‘Completed’ or ‘Rejected’

  • The transaction is registered to the ‘Online Transaction Status’

  • The relevant Quick Pay alert is forwarded to the end-user when transaction changes status to Failed or Completed or Rejected

  • The transaction is logged
