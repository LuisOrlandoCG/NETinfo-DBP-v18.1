NETinfo DBP Version 18.1 : Card Payments

Created by 1239-Mythillou Stefani, last modified by 1216-Mappoura Andrea on 13, 11, 2024

End-users can top-up their cards or make a payment to another card within the Bank. When toping up their cards, they have the option to pay the full or minimum amount (these amounts are retrieved from Core System) or specify a custom amount. The payment instruction is submitted to the Core System for execution in real time.

Card payment is considered another type of Fund Transfer, and as such, it conforms to the three steps process (i.e. submit, confirm, complete) and any other associated business rules.

Product Main Features / Business Requirements

  1. The Card Payment transaction conforms to the three step process

  2. Card payments can be triggered from the menu or from the ‘Cards’ page (if triggered from ‘Cards’ page, the card to be credited is prepopulated)

  3. Eligible debit accounts are:

    1. Accounts that fall under a Bank Product that has been configured in NETinfo DBP to allow debits (from its accounts) for the particular transaction type

    2. Accounts that end-user has Input privileges (for the particular transaction type) as defined in User Profile.

    3. Accounts under the Customer selected by end-user from the ‘Owner’ field.

  4. Option to initiate a payment to:

    1. Own Card (eligible cards must belong to a bank product that have the relevant settings in Bank Products table), or

    2. Other Card issued by the Bank (validation exists to ensure card exists)

  5. When making a payment to own card, the transfer amount can be:

    1. Custom i.e. end-user may enters any amount

    2. The minimum payment allowed for the selected card (as retrieved from Core System)

    3. The full payment for the selected card (as retrieved from Core System)

  6. Cross currency transfers are supported (configurable).

  7. Signatures are supported (configurable).

  8. Calculation of charges is supported (configurable)

  9. SCA functionality is supported (configurable).

  10. Daily limits at customer, account, user and user-account level can be enforced (configurable).

  11. Minimum and Maximum Transaction Limits can be enforced (configurable)

  12. Dedicated Alert for Card Payments is provided

  13. Disclaimers are supported (configurable)

Card Payment

The 'Card Payment' transaction conforms to the three steps process. Each step is elaborated in the sections below

Step 1: Input

The figure below ‘UI/UX - Card Payment: Input Page’, presents the ‘Card Payment’ input page. Referring to this, end-user is prompted to complete the following mandatory information:

  • Select the debit account by selecting the appropriate account from the ‘Pay From’ drop-down field

  • Select whether payment will be made to own card or to a card issued by the Bank by selecting the appropriate option in the ‘To Card’ section. In case:

    • The ‘Own Card’ is selected (this is displayed by default), the end-user is displayed with the images of their cards in a carousel and prompted to select the card of interest (upon selection the masked card number/alias and expiry date of card is displayed)

    • The ‘Other NETinfo card’ is selected, the end-user is prompted to enter the card number of the beneficiary; the cardholder name is automatically displayed in the ‘Cardholder Name’ field.

  • Select one of the payment options: custom, minimum, full:

    • If ‘custom’ is selected, end-user is prompted to enter the payment amount and define the associated currency

    • If ‘minimum’ or ‘full’ is selected, then the payment amount and currency is automatically displayed (as retrieved from Core System)

  • Provide (optionally) a reason for the card payment that will appear in their Online Transaction Status and Transaction History (if returned from the Bank’s Core System).

Once all compulsory information have been entered, end-user proceeds to the next step by pressing the ‘Submit’ button .

A disclaimer is displayed if configured

Confirm and/or Successful SCA

All validations & checks are successful, and end-user approvals are associated with the transaction (i.e. end-user has only ‘Input’ privileges)

The following actions are taken by the system:

  • The transaction changes status to ‘Pending for Authorisation’

  • The transaction is registered to the ‘Pending Authorisation’ page

  • The transaction is registered to the ‘Online Transaction Status’ page

  • A ‘Pending for Authorisation’ alert is forwarded to the approvers

  • The ‘Card Payment’ alert is forwarded to the inputter

  • The transaction is logged

End-user is navigated to Step 3: Complete.

Confirm and/or Successful SCA

All validations & checks are successful, and transaction will be executed immediately (i.e. end-user has ‘Input’ and ‘Approve’ privileges)

The following actions are taken by the system:

  • The transaction is submitted to the Core System for further processing

  • The status changes to ‘Submit’ (or ‘Failed’ if something goes wrong) and then upon response from the Core System to ‘Completed’ or ‘Rejected’

  • The transaction is registered to the ‘Online Transaction Status’

  • The ‘Card Payment’ alert is forwarded to the end-user

  • The transaction is logged

End-user is navigated to Step 3: Complete.

Step 3: Complete

Referring to the screen below (UI/UX - Complete Page), the end-user is displayed with the transaction status and the options to:

  • Go to Online Activity

  • Export the TX details in PDF

  • View Details

  • Initiate a new transfer

By pressing the ‘Done’ button, end-user is redirected to either the ‘Move Money’ landing page or the ‘Cards’ page, depending from where the card payment was triggered

A disclaimer is displayed if configured

The possible Transaction Status are:

  • Pending for Authorisation: Transaction is pending end-user authorization

  • Failed: Transaction has failed to be submitted to Core System for execution

  • Submitted: This is a transit state - transaction has been submitted to Core System but no valid response has been received

  • Pending Execution: Transaction has been submitted and Core System has responded; however processing by Core System has not completed (e.g. could be some kind of manual processing)

  • Completed: Transaction has been processed successfully by Core System

  • Rejected: Transaction has been rejected by Core System
