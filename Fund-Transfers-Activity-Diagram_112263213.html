NETinfo DBP Version 18.1 : Fund Transfers Activity Diagram

The below figure depicts the basic end-user workflow when performing a ‘Fund Transfer’ (certain activities such as checks against approval workflows at submit have been omitted for simplicity reasons; a more detailed description of the main system checks/validations/actions is presented under the diagram).

About the Activity Diagram: The Activity Diagram depicts the main activities in the workflow: boxes indicate user activity, decision boxes indicate system decision points and arrow text in brackets indicate major system activity/actions.

Step 1: Input Step

End-user first fills up the Fund Transfer Input page with all compulsory information (e.g. From Account, To Account, Transfer Amount etc.). Once submitted, the system performs the following major checks/validations/actions:

  • Performs field validations as required (e.g. size, length, type)

  • Checks whether end-user has the required access rights to the ‘Pay From’ (and ‘Pay To’ Accounts in case a Fund Transfer between Own Accounts)

  • Checks the validity of input values such as:

    • ‘Pay From’ and ‘Pay To’ accounts as/if required

    • Execution date (s)

    • Currencies (Currencies of ‘Pay From’ account - and ‘Pay To’ account as/if required- and Transfer amount)

    • Beneficiary & Intermediary bank details as/if required

    • Special codes (e.g. negotiated rate code, VO code, KPP code) as/if required

  • Checks whether SCA has been configured for the particular User Group and Transaction Type (in order to display the associated objects in the Confirm Page)

  • Calculates charges as/if configured

  • Checks whether an approval workflow (i.e. maker/checker or signatures) exists

  • Converts amounts to base currency (to be used for transaction type, end-user and customer limit checks) and debit account currency (to be used for credit check and account related limits) using the rates stored in the NETinfo DBP or the negotiated rate as/if required

  • Checks transfer amount against minimum and maximum transaction type limits

  • Checks transfer amount against daily limits (i.e. user, customer, account and user account limits) & availability of funds (available balance is retrieved from the back-end system) if:

    • End-user has approve or full access rights

    • AND there is no associated approval workflow

    • AND end-user has selected to execute the payment NOW

Step 2: Confirm Step

If all checks are successful, the system displays all details entered in step 1 in a read-only format for the end-user to confirm (double-check) before proceeding with the transfer. Other details as retrieved from the Core System are also displayed in a read only format as/if required. Upon end-user confirmation, system validates end-user as per configured SCA method as/if required and subsequently performs the following major checks/validations/actions:

  • Performs again all field and data validations as per step 1

  • Checks transfer amount against minimum and maximum transaction type limits

  • Checks transfer amount against daily limits (i.e. user, customer, account and user account limits) & availability of funds (available balance is retrieved from the back-end system) if:

    • End-user has approve or full access rights

    • AND there is no associated approval workflow

    • AND end-user has selected to execute the payment NOW

  • Upon successful validations and If transaction is connected to a end-user approval workflow (i.e. if maker/checker or signatories exist) then:

    • The transaction changes status to ‘Pending for Authorisation’

    • The transaction is registered to the ‘Pending for Authorisation’ page

    • The transaction is registered to the ‘Online Transaction Status’ page

    • A ‘Pending for Authorisation’ alert is forwarded to the approvers

    • The relevant Fund Transfer alert is forwarded to the inputter

    • The transaction is logged

  • Upon successful validations and If transaction has been scheduled to be executed at a later (specific) date then:

    • The transaction changes status to ‘Pending’ (in case where the time scheduler is provided by the Bank, the Fund Transfer instruction is submitted to the Core System for further processing)

    • The relevant Fund Transfer alert is forwarded to inputter

    • The transaction is registered to the ‘Online Transaction Status’ page

    • The transaction is logged

  • Upon successful validations and If transaction has been scheduled to be executed repeatedly (i.e. a Standing Order has been created) then:

    • A Standing Order is created and handled by the Standing Orders module thereafter (in case where Standing Orders are managed by Core System, then the instruction for Standing Order creation is submitted to the Core System for further processing)

    • The transaction is registered to the ‘Online Transaction Status’ page

    • The ‘Create Standing Order’ alert is forwarded to the inputter

    • The transaction is logged

  • Upon successful validations and if transaction is to be executed now (i.e. execution date is today and there are no approval workflows) then:

    • The transaction is submitted to the back end systems for further processing

    • The status changes to ‘Submit’ (or ‘Failed’ if something goes wrong), and then upon response from the Core System, to ‘Completed’ or ‘Rejected’

    • The transaction is registered to the ‘Online Transaction Status’ page

    • The relevant Fund Transfer alert is forwarded to the end-user when transaction changes status to Failed or Completed or Rejected

    • The transaction is logged

Step 3: Complete Step

If step 2 has been completed successfully, then the ‘Complete’ page with the associated Transaction Status, Unique Transaction Number (UTN) and Reference Number is displayed and the ‘Online Transaction Status’ page is updated as required. Additionally, depending on the design, the end-user has the option to view, export and print the transaction details in PDF format.
